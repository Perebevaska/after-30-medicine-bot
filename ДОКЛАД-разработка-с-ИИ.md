# Разработка Telegram-бота с ИИ-агентом Claude Code

### Опыт: от идеи до прода за ~9 дней

> Доклад для коллег. Каждый раздел = слайд + текст спикера.
> Все цифры — из реальной истории git и документации проекта.

---

## Слайд 1. Что построили

**Telegram-бот для напоминаний о приёме лекарств** + полноценный Telegram Mini App.

Не «hello world», а продакшн-сервис:
- Гибкое расписание приёмов (ежедневно / интервалы / дни недели / раз в месяц)
- Часовые пояса по геолокации, напоминания в local time юзера
- Статистика, PDF-экспорт для врача, серии «идеальных дней», ачивки
- «Забота» — связь двух аккаунтов: помощник видит приёмы близкого
- Соцмеханика анонимных пожеланий поддержки

**Текст спикера:** Хотел показать не игрушку, а реальный сервис с базой, очередями, авторизацией и фронтендом — потому что именно на таком масштабе видно, где ИИ-агент помогает, а где мешает.

## Слайд 3. Стек

**Backend / Bot:** Python 3.14, python-telegram-bot (async), APScheduler, ARQ + Redis (очередь от Telegram-429), PostgreSQL через psycopg3, FastAPI + Uvicorn.

**Frontend (Mini App):** Vite + TypeScript, React 19, TanStack Query, Telegram SDK, чистый CSS.

**Инфра:** Podman, systemd на VPS, Caddy (реверс-прокси + HTTPS), GitHub Actions CI/CD.

**Текст спикера:** Стек выбирал я, агент — реализовывал. Важный нюанс: агент уверенно работал с современными версиями (React 19, Python 3.14), но иногда тянул устаревшие паттерны из обучения — это ловилось ревью.

## Слайд 4. Метод разработки — главное

Не «напиши мне приложение», а **итеративный цикл с агентом как исполнителем**:

1. **Фазы.** Работа резалась на фазы (F6 геймификация, F7 забота, F10 разделение ролей, F15 пожелания…). Одна фаза = одна понятная цель.
2. **Документация-как-память.** `CLAUDE.md` — постоянный контекст для агента: архитектура, схема БД, правила. Агент читает его в начале каждой сессии.
3. **Роадмап + история.** `ROADMAP-DONE.md` (что сделано), `HISTORY.md` (какие баги и как закрыты) — агент сам ведёт летопись.
4. **Аудиты.** Отдельные проходы: безопасность (SEC), баги данных (AX), фронтенд (MF), DevOps (DV).
5. **Гейт тестами.** CI: тесты + сборка фронта + линт — жёсткий гейт перед деплоем.

**Текст спикера:** Ключевое открытие — агент без памяти беспомощен на большом проекте. Документация перестала быть «для людей» и стала рабочей памятью ИИ. CLAUDE.md важнее README.

## Слайд 5. Как выглядел рабочий цикл

```
Я: ставлю цель фазы ("связь двух аккаунтов: помощник видит приёмы")
  ↓
Агент: читает CLAUDE.md → предлагает план → пишет код + миграцию + тесты
  ↓
Я: ревью, запуск, "вот тут race condition" / "это не по TZ юзера"
  ↓
Агент: фиксит, дописывает тест на регрессию, обновляет HISTORY.md
  ↓
Коммит → CI → деплой
```

**Текст спикера:** Я ни разу не писал код руками. Моя роль сместилась: архитектор + ревьюер + заказчик. Агент — быстрые руки. Узкое место — не скорость написания, а скорость моего ревью.

## Слайд 6. Сложности — категория 1: гонки и параллелизм

Самый болезненный класс багов. Примеры из `HISTORY.md`:

- **AX1 🔴** Нет UNIQUE на `intake_log` → **дубли приёмов**. Фикс: UNIQUE-индекс + `ON CONFLICT DO UPDATE`.
- **AX2 🔴** `apply_intake_stock`: потерянное обновление при параллельных запросах → `SELECT FOR UPDATE`.
- **AX3 🔴** Dashboard: React-машина анимации **размножала карточки**. Переписали на чистый рендер (render = функция от данных).
- **AX4 🔴** Scheduler **терял напоминание** при пропуске минуты → окно догона 5 мин + дедуп.

**Текст спикера:** ИИ пишет «счастливый путь» отлично. Но конкурентность, гонки, идемпотентность — это он сам не предусматривает. Их видел человек, понимающий, что 1000 юзеров жмут кнопку одновременно.

---

## Слайд 7. Сложности — категория 2: контекст и состояние

- **AX5 🔴** Фронт считал «пора принимать» по времени **браузера**, а не по часовому поясу юзера. Перенесли расчёт на сервер.
- **AX6 🟡** Состояние напоминаний жило **в памяти процесса** — не переживало рестарт. Перенесли в Redis (TTL, fallback).
- **AX8 🟡** `REDIS_URL` захардкожен в 3 местах — агент дублировал, а не выносил в конфиг.

**Текст спикера:** Агент мыслит локально — «решить вот эту задачу здесь». Системные свойства (где живёт состояние, что переживает рестарт, единый источник конфига) — зона ответственности человека. ИИ охотно хардкодит и дублирует, если не указать обратное.

---

## Слайд 8. Сложности — категория 3: архитектурные развороты

- **Миграция SQLite → PostgreSQL** посреди проекта. После неё агент оставил **мёртвый код**, читающий старый SQLite (`broadcast.py` — баг B1), и устаревший дубль архитектуры в доках.
- **Telegram 429 (flood control).** Прямая отправка ловила лимиты → пришлось ввести очередь ARQ + Redis. До сих пор открытый риск (AX13): большой `RetryAfter` может уронить воркер.
- **O(users×meds) в планировщике** — осознанно оставленный техдолг с триггером «когда чинить».

**Текст спикера:** При крупных разворотах ИИ не подчищает за собой хвосты — остаётся мёртвый код и рассинхрон. Нужен явный аудит «что устарело после изменения». Часть техдолга мы оставили сознательно — и записали условие, когда к нему вернуться.

## Слайд 9. Сложности — категория 4: дисциплина и дрейф

- **Дрейф документации.** CLAUDE.md разрастался; приходилось его чистить/сжимать (есть `CLAUDE.original.md` — бэкап до компрессии).
- **Регрессии.** Фикс одного ломал соседнее (фронтенд-аудит ловил «мёртвый код» после рефакторингов). Спасали 227 тестов.
- **Соблазн «доверять».** Код выглядит уверенно и правдоподобно — даже когда неверен. Дешевле было проверять, чем верить.

**Текст спикера:** Главный риск — не плохой код, а *правдоподобный* код. ИИ никогда не говорит «я не уверен». Это перекладывает бремя скепсиса целиком на ревьюера.

## Слайд 10. Что сработало хорошо

- **Скорость прототипирования.** Фаза от идеи до рабочего кода — часы, не дни.
- **Тесты «бесплатно».** Агент охотно пишет тесты, если попросить — 227 штук набрались естественно.
- **Рутина и бойлерплейт.** CRUD, миграции, типы, CSS — идеально для ИИ.
- **Документация как побочный продукт.** HISTORY/ROADMAP вёл сам агент.
- **Незнакомые области.** Часовые пояса, PDF-генерация, Telegram SDK — агент быстро поднимал то, что я бы гуглил днями.

**Текст спикера:** Там, где задача хорошо определена и проверяема, ИИ даёт 5–10x. Выигрыш не в «он думает за меня», а в «он убирает трение между решением и его проверкой».

## Слайд 11. Уроки — как работать с ИИ-агентом

1. **Документация = память агента.** Веди CLAUDE.md, держи актуальным.
2. **Режь на фазы.** Маленькая проверяемая цель > «сделай всё».
3. **Тесты — не опция, а гейт.** Они ловят регрессии, которые ИИ вносит пачками.
4. **Аудиты по осям.** Отдельные проходы: безопасность, гонки, данные, мёртвый код.
5. **Человек держит системные свойства.** Конкурентность, состояние, конфиг, развороты — твоя зона.
6. **Не верь — проверяй.** Правдоподобность ≠ корректность.
7. **Записывай осознанный техдолг** с условием возврата, а не «починим потом».

## Слайд 12. Вывод

ИИ-агент сместил роль разработчика: **меньше печатать, больше думать и проверять.**

- Скорость написания перестала быть узким местом — им стали **ревью и системное мышление**.
- 228 коммитов / 227 тестов / 9 дней — реально. Но за каждым «feat» стоял человек, ловящий гонку, TZ-баг и мёртвый код.
- **ИИ — это рычаг, а не автопилот.** Усиливает того, кто понимает, что строит.

**Текст спикера (финал):** Лучшая аналогия — не «робот-программист», а очень быстрый джуниор с энциклопедической памятью, нулевым опытом эксплуатации и абсолютной уверенностью в себе. Цените его за скорость, но никогда не снимайте с него ревью.
