# План работ: Postgres + Podman + Mini App

Документ описывает поэтапный переход med-bot от текущей архитектуры (SQLite, один
процесс-бот) к продакшен-связке: **PostgreSQL** + **Podman** (несколько контейнеров) +
**Telegram Mini App** как второй фронтенд поверх общего backend.

## Базовые принципы (из предыдущих выводов)

1. **Два интерфейса — один backend, один домен, одна БД.** Бот-чат и Mini App — две
   презентации над одной доменной логикой. Логику расписания/adherence/серии **не
   дублировать** во фронте: фундамент уже изолирован в `schedule_utils.py`, `streak.py`,
   `database.py`.
2. **Postgres оправдан именно из-за второго пишущего процесса** (backend Mini App пишет
   параллельно боту). Пока второго писателя нет — SQLite достаточно.
3. **Порядок критичен: сначала мигрируем бота на Postgres и стабилизируем, потом строим
   Mini App.** Не делать миграцию БД и новый фронт одновременно.
4. **Напоминания/пуши остаются в боте** (Mini App не может разбудить пользователя).
   Mini App забирает «богатое»: редактор расписания, дашборды, статистику, запас.
5. **Планировщик (APScheduler) запускается строго в ОДНОМ процессе** (контейнер бота),
   иначе дубли напоминаний.
6. **Безопасность Mini App:** backend обязан валидировать подпись `initData` от Telegram
   (HMAC по токену бота) и брать `telegram_id` оттуда — никогда не доверять id с клиента.

## Целевая топология контейнеров (Podman)

```
                       ┌─────────────────────────┐
   Telegram  ───────►  │  bot (backend)          │  long-poll / webhook
   (чат, напоминания)  │  python-telegram-bot     │  + APScheduler (ТОЛЬКО здесь)
                       └───────────┬─────────────┘
                                   │ SQL
                       ┌───────────▼─────────────┐
   HTTPS              │  db (postgres)           │  volume: pgdata
   браузер/Telegram   └───────────▲─────────────┘
        │                          │ SQL (pool)
        │            ┌─────────────┴───────────┐
        ▼            │  api (FastAPI)           │  валидирует initData,
   ┌─────────┐  /api │  общий код с ботом       │  shared: database/schedule_utils/streak
   │ caddy   │──────►└──────────────────────────┘
   │ (proxy, │
   │  HTTPS) │  /    ┌──────────────────────────┐
   └─────────┘──────►│  miniapp (frontend)      │  статика (Vite build), отдаётся через caddy
                     └──────────────────────────┘
```

Контейнеры: **db**, **bot**, **api**, **miniapp** (статика), **caddy** (reverse proxy + авто-HTTPS).
`bot` и `api` — **один образ/репозиторий, разные команды запуска** (чтобы не расходилась
доменная логика). Объединены в один Podman pod (общая сеть), управляются через **Quadlet**
(systemd-юниты) на VPS — rootless.

## Фаза 1 — Миграция бота на Postgres (без Mini App)

**Цель:** бот работает на Postgres идентично текущему поведению. Данных мало (~52 КБ, 5
пользователей) → риск минимален, но делаем аккуратно и под тестами.

### Задачи
- **1.1 Слой соединений.** Заменить `sqlite3` на `psycopg` (3.x) + пул соединений
  (`psycopg_pool.ConnectionPool`). Сохранить контракт `get_connection()` (контекст-менеджер),
  чтобы хендлеры не трогать. Конфиг через `DATABASE_URL` (env/.env).
- **1.2 Схема (DDL).** Портировать `init_db()`/`migrate()` на Postgres:
  - `INTEGER PRIMARY KEY AUTOINCREMENT` → `GENERATED ALWAYS AS IDENTITY`.
  - Внешние ключи объявить в схеме (Postgres всегда их соблюдает).
  - Индексы — без изменений (синтаксис совместим).
  - `PRAGMA journal_mode/busy_timeout/foreign_keys` — убрать (Postgres это умеет нативно).
- **1.3 Запросы.** `?` → `%s`; upsert в `log_intake` → `INSERT ... ON CONFLICT (...) DO UPDATE`;
  `get_admin_stats` (`date(i.taken_at) = date('now')`) → Postgres-эквивалент.
- **1.4 Решения по типам (минимальный риск vs чистота):**
  - **Таймстампы.** Минимальный путь: хранить `taken_at`/`created_at` как **TEXT** в формате
    `'YYYY-MM-DD HH:MM:SS'` UTC (как сейчас) — тогда весь код парсинга (`strptime` + `pytz.utc`)
    и `local_day_bounds_utc`/`adherence_window`/`streak_window` не меняется. Генерацию времени
    перенести из `CURRENT_TIMESTAMP` **в Python** (`datetime.utcnow().strftime(...)` в параметр) —
    убирает зависимость от диалекта. Чистый путь (позже): `timestamptz`.
  - **Булевы поля** (`active`, `paused`, `caregiver_enabled`, `daily_plan_enabled`): минимальный
    путь — оставить `INTEGER` (0/1), хендлеры с `if med["paused"]` работают как есть. Чистый
    путь (позже): `boolean`.
- **1.5 Перенос данных.** Скрипт `migrate_sqlite_to_pg.py`: читает `med_bot.db`, пишет в Postgres
  (объём крошечный). Идемпотентный, с проверкой счётчиков строк.
- **1.6 Тесты.** Пайплайн тестов на Postgres:
  - Чистая логика (`test_pure`, `test_schedule_utils`, `test_streak`) — **не зависит от БД**, не трогаем.
  - DB-тесты (`test_stock_db`, `test_adherence_db`, `test_doctor_report`, `test_pause`,
    `test_delete_user_data`, `test_preset_migration`, `test_stock_intake`) — заменить временную
    SQLite на тестовую схему Postgres (testcontainers-python или отдельная тест-БД + truncate
    между тестами). Фикстуру `DB_PATH`-монки-патч заменить на `DATABASE_URL` тест-инстанса.
  - Цель: все 166 тестов зелёные на Postgres.
- **1.7 Локальный прогон** через Podman (контейнер postgres + локальный бот) для валидации.

### Критерий готовности
- 166 тестов зелёные на Postgres.
- Ручной флоу `/start → добавить → напоминание → ✅/❌ → /stats → 🔥 серия → запас → пауза → PDF`
  идентичен SQLite-версии. Часовые пояса и «сегодня» считаются корректно.
- Нет регрессий в `db_errors.log`.

### Подводные камни SQLite→Postgres (чек-лист)
- `CURRENT_TIMESTAMP` отдаёт другой формат → генерировать время в Python.
- `date('now')`, `date(col)` — sqlite-функции, портировать.
- 0/1 vs boolean — выбрать стратегию (см. 1.4) и быть последовательным.
- `INSERT OR REPLACE`/upsert → `ON CONFLICT`.
- Пустые строки vs NULL — Postgres строже к типам.
- Один пул на процесс; не открывать соединение на каждый запрос без пула.

## Фаза 2 — Контейнеризация бота (Podman + Quadlet), деплой на VPS

**Цель:** бот на Postgres работает в Podman на VPS, переживает перезагрузку. **До** Mini App.

### Задачи
- **2.1 Containerfile (backend).** Multi-stage, база `python:3.14-slim`. ⚠️ **Установить шрифты
  DejaVu** (`fonts-dejavu-core`) — fpdf2 ждёт `/usr/share/fonts/truetype/dejavu/`, иначе PDF
  (отчёт врача, экспорт) упадёт. Непривилегированный пользователь в образе.
- **2.2 Оркестрация.**
  - Локально: `podman compose` (compose-файл: postgres + bot).
  - VPS: **Quadlet**-юниты (`db.container`, `bot.container`), общий pod/сеть, volume `pgdata`,
    healthcheck'и, политика рестарта, авто-старт через systemd (rootless, `loginctl enable-linger`).
- **2.3 Секреты.** `BOT_TOKEN`, `ADMIN_ID`, `DATABASE_URL` — через Podman secrets / env-файлы
  (не в образ, не в git).
- **2.4 Бэкапы.** systemd-timer с `pg_dump` → ротация (например, 7 дней). Проверить restore.
- **2.5 Логи.** Перевести `db_errors.log`/основной лог на stdout/stderr (стандарт для контейнеров),
  собирать через `journald`/`podman logs`.

### Критерий готовности
- `bot` + `db` подняты в Podman на VPS, стартуют после reboot, напоминания приходят.
- `pg_dump` бэкап создаётся и восстанавливается.
- Решение по режиму бота зафиксировано (рекомендация: **long polling** на этом этапе — проще,
  не требует входящего HTTPS; webhook отложить до появления proxy в Фазе 5).

## Фаза 3 — Backend API для Mini App (FastAPI)

**Цель:** HTTP-API поверх той же доменной логики и той же Postgres, что и бот.

### Задачи
- **3.1 Приложение.** FastAPI в том же репозитории; импортирует `database.py`,
  `schedule_utils.py`, `streak.py`. Запуск как **отдельный контейнер/команда** (`uvicorn`).
  **Планировщик в API НЕ запускать** (он только в боте).
- **3.2 Auth.** Middleware валидации Telegram `initData` (HMAC-SHA256 по `BOT_TOKEN`), извлечение
  `telegram_id`, маппинг в внутренний `user_id`. Отклонять просроченный/поддельный initData.
- **3.3 Эндпоинты (паритет с действиями бота):**
  - чтение: лекарства, «на сегодня», статистика (week/adherence/streak), запас, план, подопечные, настройки;
  - запись: add/edit/delete лекарства, log intake (taken/skipped), пауза/возобновление, операции
    с запасом, пресеты времени, режим напоминаний, caregiver/подопечные;
  - PDF: отдавать отчёт врача/экспорт (переиспользовать `handlers/export.py` рендер в `asyncio.to_thread`).
  - Контроллеры **тонкие** — вся логика в существующих доменных функциях.
- **3.4 Безопасность/инфра API.** CORS только на домен Mini App; rate limiting; единый формат ошибок.
- **3.5 Тесты.** Валидация initData (валид/протух/подделка), happy/edge по эндпоинтам, переиспользование
  доменных функций. Пул соединений общий с ботом по схеме.

### Критерий готовности
- Любое действие, доступное в боте, доступно через API и пишет в ту же БД.
- initData-валидация обязательна; без неё — 401.
- Тесты API зелёные; нет дублирования доменной логики.

## Фаза 4 — Frontend Mini App

**Цель:** веб-интерфейс внутри Telegram для «богатых» сценариев; данные живые, общие с ботом.

### Задачи
- **4.1 Стек.** Vite + React (или Svelte) + `@telegram-apps/sdk` (или `telegram-web-app.js`),
  адаптация под `themeParams` (светлая/тёмная), TanStack Query для API, типобезопасные клиенты.
- **4.2 Экраны:**
  - **Дашборд:** «на сегодня» + 🔥 серия + краткое соблюдение, быстрые отметки.
  - **Лекарства + редактор:** заменяет мучительный многошаговый `ConversationHandler` обычной
    формой — название, дозировка, **разная дозировка**, тип расписания (daily/interval/weekdays/
    monthly), приём с пищей, мульти-слоты времени.
  - **Статистика:** графики — adherence 30 дней, серия, история; экран соблюдения.
  - **Запас:** остаток/расход/порог/прогноз, пополнение.
  - **Настройки:** пресеты времени, режим напоминаний, caregiver/подопечные.
  - **Отчёт для врача:** просмотр/скачивание PDF (через API).
- **4.3 BotFather.** Зарегистрировать Web App URL, кнопку меню; опц. inline-кнопку «Открыть
  приложение» в сообщениях бота.
- **4.4 UX-граница.** Напоминания и быстрое «принял/пропустил» — остаются в чате; Mini App —
  для просмотра/редактирования. Любое изменение в Mini App сразу видно боту (общая БД).

### Критерий готовности
- Полный CRUD лекарств и все ключевые действия доступны из Mini App.
- Данные совпадают с ботом в реальном времени (пауза в Mini App → планировщик молчит).
- Работает на iOS/Android/Desktop клиентах Telegram; корректные темы.

## Фаза 5 — Полный деплой и эксплуатация

**Цель:** вся связка на VPS, HTTPS, оба фронтенда на одной Postgres, устойчиво к перезагрузкам.

### Задачи
- **5.1 Reverse proxy.** Контейнер **Caddy** (авто-HTTPS Let's Encrypt): `/` → статика miniapp,
  `/api` → api. Домен + DNS. (Опц.: перевести бота на **webhook** теперь, когда есть HTTPS.)
- **5.2 Quadlet-юниты для всего:** `db`, `bot`, `api`, `miniapp`(или отдаётся caddy), `caddy` —
  единый pod, общая сеть, volumes, healthcheck'и, авто-рестарт, rootless.
- **5.3 CI/CD.** Сборка образов → registry (ghcr/quay) → деплой (Podman auto-update или git-pull
  на VPS). Если ручной Quadlet станет утомительным — рассмотреть **Coolify** на том же VPS.
- **5.4 Наблюдаемость.** Централизованные логи, healthcheck'и, бэкапы Postgres (из Фазы 2),
  алертинг по ошибкам. Метрики напоминаний (отправлено/подтверждено).
- **5.5 Безопасность.** Секреты вне образов, rootless least-privilege, ограничение портов
  (наружу только caddy:443), регулярные обновления базовых образов.

### Критерий готовности
- End-to-end на VPS: HTTPS, бот + Mini App на одной Postgres, переживает reboot.
- Бэкапы и восстановление проверены; деплой повторяем (одной командой/пушем).

## Сквозные риски и как закрываем

| Риск | Митигировано |
|------|--------------|
| Дубли напоминаний при нескольких процессах | APScheduler **только** в контейнере бота |
| PDF падает без шрифтов | `fonts-dejavu-core` в образе backend |
| Подделка пользователя в Mini App | Валидация `initData` HMAC, id только из него |
| Расхождение доменной логики бот/Mini App | Общий Python-код (`database`/`schedule_utils`/`streak`), тонкие контроллеры |
| Сломанные TZ/«сегодня» после миграции | Минимальный путь: TEXT-таймстампы UTC, код парсинга не трогаем |
| Контеншен/`database is locked` | Уходит сам — Postgres рассчитан на конкурентную запись |
| Миграция БД + Mini App одновременно | Строгий порядок: Фазы 1–2 (ship) → потом 3–5 |

## Рекомендуемая последовательность и параллелизм

1. **Фаза 1 → Фаза 2** — сделать и **задеплоить** (бот на Postgres в Podman) прежде всего.
   После этого — стабильная точка, можно жить.
2. **Фаза 3** (API) и **Фаза 4** (frontend) частично параллелятся: контракт API фиксируется
   первым, фронт пилится против мок-API.
3. **Фаза 5** — когда API+frontend готовы, собираем полный pod и публикуем под доменом.

## Открытые решения (зафиксировать на старте)

- Хостинг: свой VPS (Podman+Quadlet) **или** PaaS (Railway/Render) — план написан под свой VPS.
- Frontend-фреймворк: React+Vite vs Svelte.
- Таймстампы: TEXT (минимальный риск) сейчас, `timestamptz` позже — или сразу чисто.
- Булевы колонки: INTEGER сейчас vs boolean сразу.
- Режим бота: long polling (Фазы 1–2) → webhook (Фаза 5) или остаться на polling.
- Монорепо (`backend/` + `frontend/`) vs два репозитория — рекомендуется монорепо.
